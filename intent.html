WAM-Intent Namespace

Configure Phase: deterministic MCP tool configuration. The UA resolves semantic markers into tool parameters and availability signals before LLM interaction begins.

Overview

The wam-intent namespace governs the Configure phase of the Tool Lifecycle Model. Where Policy attributes (Gate phase) control which MCP mutation tools are exposed and what operations they may perform, Intent attributes configure how those tools behave when invoked — without filtering content or blocking mutations.

Intent attributes serve two distinct roles, organized into two groups:

Group A — Scoped Call Configuration

category, instruction, tone, output

These attributes compose the Scoped Call Configuration Object, which is embedded in the element's entry in the Mutable Element Manifest. When a mutation tool makes an internal LLM call (a "scoped call") on behalf of this element, this object provides the framing: how to classify the content, what directive to follow, what tone to adopt, and what format to produce.

Group B — Tool Availability Signals

function, workflow

These attributes gate exposure of the wam_invoke interaction tool. Either attribute — combined with interaction in the Effective Permission Set — causes wam_invoke to appear in the MCP tools/list for that element.

Two additional attributes influence how the element appears in the Mutable Element Manifest, but belong to neither group:

  • wam-intent-importance — Controls the ordering of elements in the Mutable Element Manifest. Higher importance elements appear earlier; lower importance elements appear later and are dropped first when token-budget limits require truncation.
  • wam-intent-entity — Enriches the manifest entry with a canonical entity URI, helping the orchestrating LLM and scoped calls resolve ambiguous named references without guessing.
Mechanistic, Not "Soft"

Intent processing is deterministic: the User Agent parses intent attributes and produces a predictable Scoped Call Configuration Object (Group A) or determines wam_invoke availability (Group B). There are no probabilistic elements. The distinction from Policy is not "hard vs soft" but where the UA acts: Policy gates and blocks; Intent configures and signals.

Attribute Group Role Example
wam-intent-category A Scoped Call Configuration "summary", "legal-disclaimer"
wam-intent-instruction A Scoped Call Configuration Free-form directive text
wam-intent-tone A Scoped Call Configuration "formal", "casual"
wam-intent-output A Scoped Call Configuration "bullet-points"
wam-intent-function B Tool Availability Signal "submit-form"
wam-intent-workflow B Tool Availability Signal "checkout-step-2"
wam-intent-importance — Manifest Ordering "critical", "background"
wam-intent-entity — Manifest Enrichment Wikidata URI

Category

The wam-intent-category attribute classifies content using a Standardized Lexicon. The value flows into the category field of the Scoped Call Configuration Object. When a mutation tool makes an internal LLM call for this element, the category is used to select an optimized system prompt from the Standardized Lexicon — helping the scoped LLM handle the content appropriately.

Example: Semantic Categories
<!-- Summary: Condensed version of longer content -->
<div wam-intent-category="summary">
  <p>TL;DR: The article explains three key points...</p>
</div>

<!-- Legal Disclaimer: Do not paraphrase -->
<footer wam-intent-category="legal-disclaimer">
  <p>Terms and conditions apply. See full terms.</p>
</footer>

<!-- Satire: Do not interpret literally -->
<article wam-intent-category="satire">
  <h2>Local Man Discovers Water Is Wet</h2>
  <p>In a stunning development...</p>
</article>

Custom Categories

If you use a category not in the Standardized Lexicon, the UA passes the value unchanged into the Scoped Call Configuration Object. The scoped LLM receives the raw string and a generic fallback framing.

Example: Custom Category
<section wam-intent-category="recipe">
  <h2>Chocolate Chip Cookies</h2>
  ...
</section>

<!-- Scoped Call Config: { category: "recipe", ... } -->

Importance

The wam-intent-importance attribute controls the ordering of elements in the Mutable Element Manifest returned by wam_build_context. Higher importance elements appear earlier in the list. When token-budget limits require truncation, elements are dropped from the tail of this ordering — so background elements are sacrificed first.

Level Manifest Position Semantic
critical First Must be addressed; cannot be dropped by truncation
high Early Prioritized over normal elements
normal Default order Standard priority (default)
low Late Supplementary; address after critical/high/normal
background Last Context only; first to be dropped if manifest is truncated
Example: Mixed Importance Levels
<article>
  <!-- Critical: appears first in wam_build_context manifest -->
  <div wam-intent-importance="critical">
    <p><strong>WARNING:</strong> Server maintenance at midnight.</p>
  </div>

  <!-- Normal: standard position in manifest -->
  <p>Regular article content goes here...</p>

  <!-- Background: last in manifest, dropped first on truncation -->
  <aside wam-intent-importance="background">
    <p>Historical note: This feature was added in v2.3...</p>
  </aside>
</article>
Ordering, Not Filtering

Importance affects the order elements appear in the manifest, not whether they are included. All mutable elements appear in the manifest regardless of importance level; background elements simply appear last and are the first candidates for removal when token budgets require it.

Entity

The wam-intent-entity attribute links text to a canonical entity URI (such as a Wikidata entry). The URI is included in the element's entry in the Mutable Element Manifest, helping the orchestrating LLM and any scoped calls resolve ambiguous named entities without relying on the LLM's parametric knowledge alone.

Example: Entity Disambiguation
<p>
  <!-- Manifest entry: intent.entity = "https://www.wikidata.org/wiki/Q312" -->
  <span wam-intent-entity="https://www.wikidata.org/wiki/Q312">
    Apple
  </span>
  announced new products today.
</p>

<p>
  <!-- Different entity: the fruit -->
  I ate an
  <span wam-intent-entity="https://www.wikidata.org/wiki/Q89">
    apple
  </span>
  for lunch.
</p>

Benefits:

  • Disambiguates homonyms (Apple Inc. vs. the fruit) without relying on LLM guesses
  • Provides a stable, structured reference independent of text phrasing
  • Enables precise entity-based queries and cross-document linking
  • Improves accuracy for any mutation or scoped call that touches this element

Instruction

The wam-intent-instruction attribute is a Group A field that provides a free-form directive included directly in the Scoped Call Configuration Object. When a mutation tool makes an internal LLM call, the instruction field has the highest precedence — the scoped LLM uses it verbatim as the primary system prompt fragment, overriding what the category lexicon would otherwise supply.

Example: Custom Instructions
<!-- Scoped calls receive: { instruction: "Translate all technical jargon into plain English", ... } -->
<div wam-intent-instruction="Translate all technical jargon into plain English">
  <p>
    The API utilizes RESTful architecture with OAuth 2.0
    authentication and returns JSON-formatted responses.
  </p>
</div>

<!-- Format-specific instruction -->
<div wam-intent-instruction="Summarize in exactly 3 bullet points">
  <p>Long article content here...</p>
</div>
Precedence Within Group A

When wam-intent-instruction is present, the UA uses it as the primary system prompt fragment for any scoped LLM call. Other Group A fields (category, tone, output_format) are still resolved and appended after it — instruction does not suppress them, it just takes precedence over the lexicon category prompt.

Configure Phase

The Configure phase is the second step in the WAM Tool Lifecycle Model, between the Gate phase (policy enforcement) and the Record phase (provenance). The UA resolves intent attributes into two outputs:

Group A: Scoped Call Configuration Object

Group A attributes (category, instruction, tone, output) compose the Scoped Call Configuration Object. This object is embedded in the element's entry in the Mutable Element Manifest and passed to any mutation tool that accepts an instruction parameter instead of a value parameter.

A mutation tool receiving an instruction parameter performs a scoped LLM call: it fetches the element's content (filtered by input policy), assembles a scoped system prompt from the Configure phase object, calls an internal LLM, and uses the response as the mutation value.

Scoped Call Configuration Object (JSON)
// Element attributes:
// wam-intent-category="api-spec"
// wam-intent-instruction="Simplify for a junior developer audience"
// wam-intent-tone="casual"
// wam-intent-output="bullet-points"

// Resolved Scoped Call Configuration Object (in Mutable Element Manifest):
{
  "category": "api-spec",
  "instruction": "Simplify for a junior developer audience",
  "tone": "casual",
  "output_format": "bullet-points"
}
Scoped LLM Calls

Not all mutation tool invocations trigger a scoped call. Only tool calls that supply an instruction parameter (rather than a concrete value) initiate an internal LLM call. The Scoped Call Configuration Object provides the framing for that internal call, ensuring authors control how the LLM operates on their element even when they don't control the mutation value directly.

Group B: wam_invoke Availability

Group B attributes (function, workflow) gate exposure of the wam_invoke interaction tool. The rule is:

wam_invoke Exposure Rule

wam_invoke appears in tools/list for an element if and only if:

  1. The element's Effective Permission Set contains interaction (from wam-policy-output), AND
  2. The element declares wam-intent-function OR wam-intent-workflow (or both).

Either Group B attribute satisfies condition 2 (OR semantics). The policy condition 1 always takes precedence — no interaction permission means no wam_invoke, regardless of Group B.

When wam_invoke is exposed, its available parameters reflect what Group B attributes are declared:

Declared Attributes wam_invoke Parameters Available
wam-intent-function="submit-form" only action: "submit-form"
wam-intent-workflow="checkout-step-2" only stage: "checkout-step-2"
Both attributes declared action: "submit-form", stage: "checkout-step-2"

Tone & Output Format (Group A)

wam-intent-tone and wam-intent-output are Group A attributes that populate fields in the Scoped Call Configuration Object. They direct how a scoped LLM call should style its response.

wam-intent-tone

Populates the tone field in the Scoped Call Configuration Object.

<div wam-intent-tone="formal">...

Scoped Call Config: { ..., "tone": "formal" }

wam-intent-output

Populates the output_format field in the Scoped Call Configuration Object, specifying the desired response format for scoped LLM calls.

<div wam-intent-output="bullet-points">...

Scoped Call Config: { ..., "output_format": "bullet-points" }

Function & Workflow (Group B)

wam-intent-function and wam-intent-workflow are Group B attributes. They do not configure scoped LLM calls — they control whether wam_invoke is available for an element and what parameters it accepts.

wam-intent-function

Describes what an interactive control does. When present (together with interaction output permission), the UA exposes wam_invoke with an action parameter constrained to this value. 

<button
  wam-policy-output="interaction"
  wam-intent-function="submit-form">Submit</button>

<!-- wam_invoke is now available with: { action: "submit-form" } -->

wam-intent-workflow

Indicates the current stage in a multi-step process. When present (together with interaction output permission), the UA exposes wam_invoke with a stage parameter constrained to this value. 

<section
  wam-policy-output="interaction"
  wam-intent-workflow="checkout-step-2">...

<!-- wam_invoke is now available with: { stage: "checkout-step-2" } -->
interaction Policy Required

Group B attributes alone are not sufficient to expose wam_invoke. The element's output policy must also include interaction (or mutable, which includes all tokens). Without interaction in the Effective Permission Set, wam_invoke is never offered even if both Group B attributes are declared.
