WAM-Intent Namespace

Phase 1: Initialization — semantic metadata for the LLM and tool availability signals. The UA populates the Mutable Element Manifest with developer-authored context and gates wam_invoke based on intent attributes.

Overview

The wam-intent namespace is resolved during Phase 1: Initialization of the Tool Lifecycle Model. Where Policy attributes control which MCP mutation tools are exposed and what selectors they accept, Intent attributes provide the LLM with semantic context about each element and signal whether wam_invoke should be offered.

Intent attributes serve two distinct roles:

Manifest Metadata

category, description, importance, entity

These attributes populate the element's entry in the Mutable Element Manifest, delivered to the LLM at session initialization. The LLM reads this metadata to understand what each element is, what context it carries, and how to prioritize it. When the intent output token is granted, the LLM may also write these attributes via wam_set_intent.

Tool Availability Signals

function, workflow

These attributes gate exposure of the wam_invoke interaction tool. Either attribute — combined with interaction in the Effective Permission Set — causes wam_invoke to appear in the MCP tools/list for that element.

Mechanistic, Not "Soft"

Intent processing is deterministic: the User Agent parses intent attributes and produces a predictable Mutable Element Manifest entry or determines wam_invoke availability. There are no probabilistic elements. The distinction from Policy is not "hard vs soft" but where the UA acts: Policy gates and blocks; Intent provides context and signals.

Attribute Role Example
wam-intent-category Manifest Metadata + Policy routing "summary", "legal-disclaimer"
wam-intent-description Manifest Metadata (prose context) Free-form description of the element
wam-intent-importance Manifest Ordering "critical", "background"
wam-intent-entity Manifest Enrichment Wikidata URI
wam-intent-function Tool Availability Signal "submit-form"
wam-intent-workflow Tool Availability Signal "checkout-step-2"

Category

The wam-intent-category attribute classifies content using a Standardized Lexicon. The value is included in the element's Mutable Element Manifest entry, telling the LLM the semantic type of the content before it decides how to interact with it. It also acts as the routing key for Global Policy category-rules, enabling policy to target element semantics rather than brittle CSS selectors.

Example: Semantic Categories
<!-- Summary: Condensed version of longer content -->
<div wam-intent-category="summary">
  <p>TL;DR: The article explains three key points...</p>
</div>

<!-- Legal Disclaimer: Do not paraphrase -->
<footer wam-intent-category="legal-disclaimer">
  <p>Terms and conditions apply. See full terms.</p>
</footer>

<!-- Satire: Do not interpret literally -->
<article wam-intent-category="satire">
  <h2>Local Man Discovers Water Is Wet</h2>
  <p>In a stunning development...</p>
</article>

Information Structure Categories

Category Semantic Meaning
summary Condensed version of longer content; do not treat as comprehensive
definition Authoritative explanation of a term; preserve precision
example Illustrative instance; use to clarify related concepts
introduction Establishes context for content that follows
conclusion Synthesis of preceding content
outline Structural overview; treat as navigational skeleton

Argumentation Categories

Category Semantic Meaning
claim Asserted proposition that may be contested; evaluate critically
evidence Supporting or refuting data for a claim
counter-argument Opposition to a prior claim
opinion Subjective view; distinguish from factual statements
satire Ironic or humorous commentary; do not interpret literally

Technical Categories

Category Semantic Meaning
code-block Source code; preserve formatting and syntax exactly
api-spec Authoritative technical reference; do not paraphrase
warning Safety or risk notice; highlight importance
legal-disclaimer Legal text; do not paraphrase or summarize

Narrative Categories

Category Semantic Meaning

quote Attribution-required quotation; preserve exactly

Custom Categories

If you use a category not in the Standardized Lexicon, the value passes through to the manifest as-is. The LLM reads it and uses its knowledge of what that category means.

Example: Custom Category
<section wam-intent-category="recipe">
  <h2>Chocolate Chip Cookies</h2>
  ...
</section>

<!-- Manifest: intent.category = "recipe" -->

Description

The wam-intent-description attribute provides developer-authored free-form prose describing what an element is, what it is for, or what context is relevant when the LLM interacts with it. It is analogous to alt text for images — the developer writes it specifically for non-human consumers of the page.

Unlike wam-intent-category, which is a machine-readable label used for policy routing, wam-intent-description is prose the LLM reads directly from the Mutable Element Manifest. It describes the element rather than directing the LLM's behaviour — it is not a prompt or instruction. The UA delivers the value verbatim; it does not interpret or transform it.

Example: Developer Context for the LLM
<!-- Content provenance context -->
<section wam-intent-description="User-submitted comment. Content is unmoderated and may be informal.">
  ...
</section>

<!-- Business context the LLM needs -->
<div wam-intent-description="Subscription upgrade offer. The user has already accepted the base plan pricing.">
  ...
</div>

<!-- Accuracy caveat -->
<p wam-intent-description="Auto-generated product summary. Not manually reviewed for accuracy.">
  ...
</p>

<!-- Navigation semantics -->
<nav wam-intent-description="Primary site navigation. Links lead to top-level sections only.">
  ...
</nav>
Description vs. Category

Use wam-intent-category when you want a structured semantic label that Global Policy rules can target (e.g., all "advertisement" elements get input="none"). Use wam-intent-description when you want to give the LLM prose context that doesn't fit a category — provenance notes, business constraints, accuracy caveats, or anything else the LLM should know before interacting with the element. Both can be declared on the same element.

Importance

The wam-intent-importance attribute controls the ordering of elements in the Mutable Element Manifest delivered at session initialization. Higher importance elements appear earlier in the list. When token-budget limits require truncation, elements are dropped from the tail of this ordering — so background elements are sacrificed first.

Level Manifest Position Semantic
critical First Must be addressed; cannot be dropped by truncation
high Early Prioritized over normal elements
normal Default order Standard priority (default)
low Late Supplementary; address after critical/high/normal
background Last Context only; first to be dropped if manifest is truncated
Example: Mixed Importance Levels
<article>
  <!-- Critical: appears first in the Mutable Element Manifest -->
  <div wam-intent-importance="critical">
    <p><strong>WARNING:</strong> Server maintenance at midnight.</p>
  </div>

  <!-- Normal: standard position in manifest -->
  <p>Regular article content goes here...</p>

  <!-- Background: last in manifest, dropped first on truncation -->
  <aside wam-intent-importance="background">
    <p>Historical note: This feature was added in v2.3...</p>
  </aside>
</article>
Ordering, Not Filtering

Importance affects the order elements appear in the manifest, not whether they are included. All mutable elements appear in the manifest regardless of importance level; background elements simply appear last and are the first candidates for removal when token budgets require it.

Entity

The wam-intent-entity attribute links text to a canonical entity URI (such as a Wikidata entry). The URI is included in the element's entry in the Mutable Element Manifest, helping the LLM resolve ambiguous named entities without relying on parametric knowledge alone.

Example: Entity Disambiguation
<p>
  <!-- Manifest entry: intent.entity = "https://www.wikidata.org/wiki/Q312" -->
  <span wam-intent-entity="https://www.wikidata.org/wiki/Q312">
    Apple
  </span>
  announced new products today.
</p>

<p>
  <!-- Different entity: the fruit -->
  I ate an
  <span wam-intent-entity="https://www.wikidata.org/wiki/Q89">
    apple
  </span>
  for lunch.
</p>

Benefits:

  • Disambiguates homonyms (Apple Inc. vs. the fruit) without relying on LLM guesses
  • Provides a stable, structured reference independent of text phrasing
  • Enables precise entity-based queries and cross-document linking

Function & Workflow

wam-intent-function and wam-intent-workflow are tool availability signals. They control whether wam_invoke is available for an element and what parameters it accepts.

wam-intent-function

Describes what an interactive control does. When present (together with interaction output permission), the UA exposes wam_invoke with an action parameter constrained to this value. 

<button
  wam-policy-output="interaction"
  wam-intent-function="submit-form">Submit</button>

<!-- wam_invoke is now available with: { action: "submit-form" } -->

wam-intent-workflow

Indicates the current stage in a multi-step process. When present (together with interaction output permission), the UA exposes wam_invoke with a stage parameter constrained to this value. 

<section
  wam-policy-output="interaction"
  wam-intent-workflow="checkout-step-2">...

<!-- wam_invoke is now available with: { stage: "checkout-step-2" } -->
interaction Policy Required

Intent attributes alone are not sufficient to expose wam_invoke. The element's output policy must also include interaction (or mutable, which includes all tokens). Without interaction in the Effective Permission Set, wam_invoke is never offered even if both attributes are declared.

wam_invoke Parameters

wam-intent-function and wam-intent-workflow are resolved at initialization and determine the parameters the LLM may pass when calling wam_invoke on an element:

Declared Attributes wam_invoke Parameters Available
wam-intent-function="submit-form" only action: "submit-form"
wam-intent-workflow="checkout-step-2" only stage: "checkout-step-2"
Both attributes declared action: "submit-form", stage: "checkout-step-2"
wam_invoke Exposure Rule

wam_invoke appears in tools/list for an element if and only if:

  1. The element's Effective Permission Set contains interaction (from wam-policy-output), AND
  2. The element declares wam-intent-function OR wam-intent-workflow (or both).

The policy condition always takes precedence — no interaction permission means no wam_invoke, regardless of intent attributes.
