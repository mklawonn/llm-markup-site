Tool Lifecycle Model

How the User Agent mediates every LLM interaction with the page through MCP tools.

Overview

The Tool Lifecycle Model is the core enforcement mechanism of Web Agent Markup. Every LLM action on a document — reading content, modifying elements, invoking controls — must occur through a defined WAM MCP tool. The User Agent must not allow the LLM to observe or modify the document by any other mechanism.

WAM governs two phases of every LLM session, one at the beginning and one at every mutation. The three WAM namespaces map onto these phases:

  • Policy attributes determine which tools are offered and what they can touch — resolved at initialization
  • Intent attributes characterize each mutable element in the manifest — also resolved at initialization
  • Provenance attributes are written by the UA atomically with each tool call — the Record phase
No Direct DOM Access

The LLM never has direct access to the DOM. Its view of the page comes entirely from the filtered HTML fragment the UA delivers at initialization — wam-policy-input controls what appears in that fragment, and the same filtering applies when the LLM calls wam_read_element. Changes can only be made through mutation tools gated by wam-policy-output. This is what makes WAM policy enforcement deterministic rather than probabilistic.

The Two Phases

A conforming User Agent implements an internal MCP server that sits between the LLM and the page. The LLM interacts exclusively through tool calls. WAM enforces two phases; the LLM's reasoning between them is outside the standard's scope:

1. Initialization
wam-policy-* & wam-intent-*

Which tools are offered?
What selectors are in each enum?
What content is visible?
What metadata does each element carry? 		→ 2. Record
wam-provenance-*

Write the Provenance Ledger entry.
Update wam-provenance-operation.
Then apply the DOM change.
Repeat for every tool call.

Initialization runs once, before the LLM session begins. Record runs for every tool call that modifies the document — mutation tools and wam_invoke. The LLM makes as many calls as needed; each one is an independent Record phase execution. There is no unrecorded LLM-initiated change.

Phase 1: Initialization

Before the LLM session begins, the UA runs context discovery and delivers three things to the LLM as part of session setup:

  1. A filtered HTML fragment — the page after wam-policy-input has been applied. Elements with input="none" are omitted entirely; elements with input="structure" have their text replaced with [REDACTED].
  2. A composed system prompt — assembled from wam-intent-* attributes across the page.
  3. The Mutable Element Manifest — a structured list of every element for which at least one mutation or interaction tool is available, including which tools apply and what intent metadata each element carries (category, description, importance, entity).

Alongside the context, the UA builds tools/list. For each mutation and interaction tool, the UA collects the selectors of every element whose policy permits it. If at least one exists, the tool appears in tools/list with its selector parameter constrained to a JSON Schema enum of exactly those selectors. A tool with no permitted elements is omitted entirely. The LLM cannot form a valid call targeting an element the tool does not permit.

wam-intent-* attributes are fully resolved at this stage. They do not gate tool calls at call time — their work is done here, populating the manifest and determining wam_invoke availability.

tools/list is stable for the duration of the session. If a DOM modification invalidates the current selectors, the UA re-runs context discovery and rebuilds tools/list before accepting further tool calls.

Visibility Is a Prerequisite for Mutability

An element with wam-policy-input="none" is invisible to the LLM regardless of its wam-policy-output value. It is absent from the filtered HTML and from the Mutable Element Manifest, so its selector never appears in any tool's enum. The LLM cannot discover the element exists and therefore cannot target it.

Phase 2: Record

Every tool call that modifies the document is executed atomically with a Provenance Ledger write. The UA writes the ledger entry and appends a <layer>:<explanation> token to the element's wam-provenance-operation attribute before applying the DOM change. The ledger write and DOM application are atomic: either both occur or neither does. There are no unrecorded LLM-initiated changes.

Field Description
id, timestamp, target Entry identity and location
layer content, style, interaction, topology, data, or intent
explanation Why the LLM made this change
originalValue, newValue Before and after values
orchestratingModelId The model that issued the tool call
generatingModelId The model that produced the content when the UA delegated to an external service; omitted when the orchestrating model provided the value directly

The UA executes in strict order: Verify the selector resolves and the session is current → Execute the mutation in memory → Write the Provenance Ledger entry → Append the <layer>:<explanation> token to wam-provenance-operation → Apply to the live DOM → Return success.

This sequence applies equally to mutation tools and to wam_invoke. An invoked control is recorded with layer interaction before the invocation is dispatched to the page.

Worked Example: Annotation Tool Call

Here is a complete walkthrough of a single tool call passing through both phases.

Setup

Page Markup
<article id="news-story"
         wam-policy-input="all"
         wam-policy-output="annotation style"
         wam-intent-category="summary"
         wam-intent-description="Central bank rate cut story — wrap key economic terms in <mark> elements when annotating"
         wam-provenance-source="https://news.example.com/story/42">
  <p>
    The central bank announced a 25 basis point rate cut on Tuesday,
    citing weakening consumer spending and a cooling labor market.
  </p>
</article>

A user asks their browser agent: “Highlight the key economic terms in this article.”

Step 1: Initialization (UA-Internal)

Before the LLM session begins, the UA runs context discovery. The input policy is all, so the full article text is visible. The output policy grants annotation and style, so the UA includes wam_annotate and wam_apply_style in tools/list with #news-story in each selector enum. The UA delivers the filtered HTML, composed system prompt, and Mutable Element Manifest to the LLM:

Manifest Entry for #news-story
{
  "selector": "#news-story",
  "wam_id": "news-story",
  "available_tools": ["wam_annotate", "wam_apply_style"],
  "intent": {
    "category": "summary",
    "description": "Central bank rate cut story — wrap key economic terms in <mark> elements when annotating"
  },
  "provenance": {
    "source": "https://news.example.com/story/42"
  }
}

The LLM now knows: it can annotate and style this element, but not rewrite its text. It also knows the element's semantic category and the developer's description of what annotation is appropriate.

Step 2: Tool Call

Guided by the manifest context and the user's request, the LLM composes the annotated HTML and calls wam_annotate with the result directly:

LLM Tool Call
wam_annotate({
  selector: "#news-story",
  value: "<p>The <mark>central bank</mark> announced a <mark>25 basis point rate cut</mark>...</p>",
  explanation: "highlighted-economic-terms"
})

The UA validates the supplied value is non-destructive: the textContent of the element is unchanged. Only inline wrapper elements have been added.

Step 3: Record Phase

Before applying the DOM change, the UA writes a Provenance Ledger entry:

Ledger Entry
{
  "id": "entry-001",
  "timestamp": "2025-07-11T14:30:00Z",
  "target": "#news-story",
  "layer": "content",
  "explanation": "highlighted-economic-terms",
  "originalValue": "<p>The central bank announced...",
  "newValue": "<p>The <mark>central bank</mark> announced...",
  "orchestratingModelId": "claude-opus-4-6"
}

The UA appends content:highlighted-economic-terms to the element's wam-provenance-operation attribute, then applies the mutation to the live DOM.

Step 4: DOM Applied

The mutation is applied to the live DOM. The element now contains <mark>-wrapped terms. The user sees highlighted economic terminology, the original text is unchanged, and the provenance is recorded.

Both Phases in Action

Initialization determined which tools were available and delivered the element's intent metadata to the LLM (summary category, developer description). Record ensured the mutation was committed atomically with its provenance entry. The LLM's reasoning — reading the context and deciding what annotation to produce — happened between the two phases, outside WAM's scope.
